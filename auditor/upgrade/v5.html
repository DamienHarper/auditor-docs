Version: 4.x

Upgrading to 5.x from 4.x

New audit table schema — ULID transactions, unified blame JSON, structured diffs

This document describes the backward incompatible changes introduced in auditor 5.0 and how to adapt your code.

⚠️ Overview​

Version 5 replaces the legacy audit table schema (v1) with a new unified schema (v2). The main changes are:

What changedv1 (4.x)v2 (5.x)
Transaction groupingtransaction_hash VARCHAR(40) SHA1transaction_id CHAR(26) ULID
Blame context4 separate columnssingle blame JSON column
Diffs formatad-hoc per operation typeunified {source, changes} envelope
Schema marker(none)schema_version SMALLINT

Existing data is preserved. The audit:schema:migrate command migrates your data safely. New installations get the v2 schema automatically.

📋 Migration Steps​

1️⃣ Update dependencies​

composer require damienharper/auditor:^5.0
composer require damienharper/auditor-doctrine-provider:^5.0

2️⃣ Update your code (see breaking changes below)​

3️⃣ Migrate your audit tables​

CAUTION

Back up your database before running the migration.

# Preview what will be executed
php bin/console audit:schema:migrate --dump-sql

# Recommended: migrate schema and convert all legacy data
php bin/console audit:schema:migrate --force --convert-all

For large databases, process in batches to estimate time first:

# Dry-run on 1 000 transactions to measure speed
php bin/console audit:schema:migrate --force --convert-all --limit=1000

# Then run fully
php bin/console audit:schema:migrate --force --convert-all

See the Schema Management guide for all audit:schema:migrate options.

4️⃣ Run audit:schema:update​

php bin/console audit:schema:update --force

NOTE

audit:schema:update refuses to run if any audit table still has the legacy v1 schema. Always run audit:schema:migrate first.

💥 Breaking Changes​

Entry model — transactionHash removed, transactionId added​

The $entry->transactionHash property (SHA1 string) has been replaced by $entry->transactionId (ULID string).

// Before (4.x)
$hash = $entry->transactionHash;

// After (5.x)
$id = $entry->transactionId;

NOTE

During the migration window (before audit:schema:migrate has been run), $entry->transactionId transparently falls back to the legacy transaction_hash value so audit streams remain accessible without downtime.

Entry model — individual blame properties replaced by blame array​

The four individual blame properties have been consolidated into a single blame array with convenience accessors.

Before (4.x)After (5.x)
$entry->username$entry->username (unchanged)
$entry->userFqdn$entry->userFqdn (unchanged)
$entry->userFirewall$entry->userFirewall (unchanged)
$entry->ip$entry->ip (unchanged)
(new)$entry->blame → full ['username', 'user_fqdn', 'user_firewall', 'ip'] array

The convenience accessors (username, userFqdn, userFirewall, ip) still work unchanged — they now delegate to the blame JSON column.

Entry model — getDiffs() now returns changes sub-array for v2 rows​

For rows written with schema_version = 2, getDiffs() returns only the changes map, not the full diffs envelope. Legacy rows (schema_version = 1) return the raw array unchanged, so existing consumers keep working.

// Both schema versions: field-level changes (no code change needed for most consumers)
$diffs = $entry->getDiffs();
// ['title' => ['old' => 'Before', 'new' => 'After'], ...]

New methods available for v2 rows:

// Source entity metadata (schema_version >= 2 only, null for legacy rows)
$source = $entry->getDiffSource();
// ['id' => '42', 'class' => 'App\Entity\Post', 'label' => 'My post', 'table' => 'posts']

// Target entity for associate/dissociate (schema_version >= 2 only)
$target = $entry->getDiffTarget();
// ['id' => '5', 'class' => 'App\Entity\Tag', 'label' => 'php', 'table' => 'tags']

LifecycleEvent payload — blame_user* / ip keys replaced by blame_* keys​

The payload dispatched on LifecycleEvent no longer contains the v1 keys. Update any listeners that read blame or transaction data.

Before (4.x)After (5.x)
$payload['transaction_hash']$payload['transaction_id']
$payload['blame_user']$payload['blame_user'] (kept for compatibility)
$payload['blame_user_fqdn']$payload['blame_user_fqdn'] (kept for compatibility)
$payload['blame_user_firewall']$payload['blame_user_firewall'] (kept for compatibility)
$payload['ip']$payload['ip'] (kept for compatibility)
// Before (4.x)
$hash = $payload['transaction_hash'];

// After (5.x)
$id = $payload['transaction_id'];

audit:schema:update now blocks on v1 tables​

audit:schema:update refuses to run if any audit table still has the legacy v1 schema (presence of transaction_hash column). It will print an error listing the affected tables and ask you to run audit:schema:migrate first.

This guard prevents accidental data loss: running audit:schema:update on a v1 table would silently drop transaction_hash and blame_* columns before the data could be converted.

🆕 New Features​

audit:schema:migrate command​

A new command handles the one-time v1 → v2 data migration:

# Full migration with all data conversions (recommended)
php bin/console audit:schema:migrate --force --convert-all

# Run conversions independently (any order)
php bin/console audit:schema:migrate --force --convert-diffs
php bin/console audit:schema:migrate --force --convert-transaction-hash

See Schema Management for details.

Entry::getDiffSource() and Entry::getDiffTarget()​

Two new methods expose the source and target metadata blocks from the v2 diffs envelope.

Entry::$blame property​

New virtual property returning the full blame context as an array: ['username', 'user_fqdn', 'user_firewall', 'ip'].

Entry::$schemaVersion property​

New int property exposing the schema_version DB column value. Useful for conditional logic when processing a mix of legacy and migrated rows.

🗄️ Audit Table Schema Changes​

Removed columns​

ColumnReplaced by
transaction_hash VARCHAR(40)transaction_id CHAR(26)
blame_user VARCHAR(255)blame JSON
blame_user_fqdn VARCHAR(255)blame JSON
blame_user_firewall VARCHAR(100)blame JSON
ip VARCHAR(45)blame JSON

Added columns​

ColumnTypeDescription
schema_versionSMALLINTRow format marker: 2 for v5+, 1 for unconverted rows
transaction_idCHAR(26)ULID grouping changes from the same flush
blameJSON{username, user_fqdn, user_firewall, ip}

New composite indexes​

IndexColumns
object_id_type_{hash}_idx(object_id, type)
blame_id_created_at_{hash}_idx(blame_id, created_at)

🔧 Troubleshooting​

audit:schema:update reports v1 tables​

Run audit:schema:migrate --force --convert-all first, then retry audit:schema:update.

$entry->transactionId is null after migration​

The row was written before migration and --convert-transaction-hash was not run. Either re-run audit:schema:migrate --force --convert-transaction-hash, or accept that pre-migration rows have no ULID (they will still display using the legacy hash value as a fallback).

Diffs look different after migration​

getDiffs() for v2 rows returns the changes sub-array (field-level only). If you previously accessed @source metadata from getDiffs(), use getDiffSource() instead.
